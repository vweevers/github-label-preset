1. Alternatively this kind of product flow can be represented on a kanban-style Project Board, per repository or organization. This way labels describe issue type or unforeseen status, while the horizontal timeline of a project board describes the mandatory stages that each issue goes through before public release.

Example combinations

Breaking feature enhancement major

New logo to discuss in meeting design agenda

Illegal combinations

Feature as patch enhancement patch
Status unclear blocked blocker
Design does not mean system design 1 backend design
Nor is scaling about image size scaling design
Or marketing a production priority marketing production
Or major about importance 2 bug major
1. Instead use labels like discussion (if an open question) or docs (if for a Product Requirements Document).
2. To avoid this ambiguity, some prefer labels like semver-major. But longer labels mean more visual clutter; I prefer short labels accompanied by a clear set of guidelines.
